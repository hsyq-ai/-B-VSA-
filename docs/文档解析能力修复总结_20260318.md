# 文档解析能力修复总结（2026-03-18）

## 一、问题背景

- 现象：用户在会话中上传 PDF 后，AI 多次反馈“下载失败/空文件”，无法分析附件。
- 影响：会话附件虽然已上传入库，但 Agent 在推理链路中拿不到可解析正文。

## 二、根因梳理

### 1) 上传成功但解析失败

- 附件上传接口会成功写入数据库和磁盘文件：
  - 元数据：`chat_files.db`
  - 文件路径：`~/.copaw/chat_files/{user_id}/{session_id}/...`
- 失败发生在 Agent 的“二次取文件”阶段，不在上传阶段。

### 2) 为什么会“上传后还下载”

- 当前架构中，用户消息里的 `file` block 保存的是下载地址（URL）或来源描述，不直接内嵌全文。
- Agent 在 `reply` 前会执行文件块处理，把附件转换成可用本地路径，再继续推理。

### 3) 失败点（按时间演进）

- 早期失败：`/api/chat-files/...` 为相对路径，下载器无法直接请求。
- 后续失败：外网映射 URL 变化导致下载到空文件。
- 最新定位：同一台机器上存在“不同进程/不同启动方式”，导致线上实际处理请求的进程未加载最新修复代码；同时在部分沙箱环境中 `~/.copaw` 被挂载为只读，导致直接用该目录启动会在写日志时崩溃。

## 三、本次代码改动

## 1) 增强 URL 归一化，优先回源本地 API

- 文件：`src/copaw/agents/utils/file_handling.py`
- 关键点：
  - 对相对路径 `/api/...` 自动补全为 `http://127.0.0.1:{port}/api/...`
  - 对 `http(s)://.../api/chat-files/...` 也改写到本机 API（保留 query/fragment），避免依赖临时外网映射域名

## 2) 建立“确定性文档解析链路”

- 文件：`src/copaw/agents/utils/message_processing.py`
- 新增能力：
  - 对下载后的 `file` 自动抽取正文并写回同条消息上下文文本块
  - PDF 解析优先级：`pypdf` -> `pdfplumber` -> `pdftotext`
  - DOCX 解析优先级：`python-docx` -> `pandoc`
  - 纯文本类文件（txt/md/csv/log/json/yaml）直接读取
  - 自动截断超长正文，避免上下文爆炸（20000 字符）
  - 兼容历史消息：当 `file` block 的 `source` 已经是本地路径字符串时，仍能被识别并进入同一套处理流程
- 结果：模型不再只看到“文件路径”，而是稳定拿到正文片段用于分析。

## 3) 前端附件 URL 策略回调

- 文件：`console/src/pages/Chat/index.tsx`
- 处理：
  - `withAccessToken` 不再强行拼接 `window.location.origin`（保持后端返回的 URL 形态，仅追加 token）
  - 避免把附件请求绑定到临时外网端口，和后端“优先回源本机 API”策略配合使用

## 4) 依赖声明补齐

- 文件：`pyproject.toml`
- 新增依赖：
  - `pypdf>=5.4.0`
  - `pdfplumber>=0.11.5`
  - `reportlab>=4.3.1`
  - `python-docx>=1.1.2`

## 5) 启动方式与运行目录调整（确保加载新代码）

- 原问题：旧进程仍在服务，且其运行环境未加载仓库最新代码，导致修复无效。
- 新做法：
  - 使用 `PYTHONPATH={repo}/src` 方式启动，确保直接加载仓库代码（避免指向旧安装包代码）。
  - 将 `COPAW_WORKING_DIR` 指向可写目录（避免在沙箱/受限环境下对 `~/.copaw` 写入失败）。
  - 将 `~/.copaw` 复制一份到工作区目录以保留原会话数据（`chat_files.db / chat_files / sessions / config.json` 等）。

## 六、给其他模型的执行建议

- 若用户反馈“上传失败”，先区分是：
  - 上传接口失败
  - 还是上传成功但 Agent 取文件/解析失败
- 优先检查日志关键字：
  - `unknown url type`
  - `Downloaded file is empty`
  - `Failed to process file block`
- 若修复代码已合入但线上仍失败，优先确认：
  - 实际服务进程是否加载了最新代码（进程启动方式/安装路径/PYTHONPATH）
  - 运行目录是否可写（`WORKING_DIR/copaw.log` 是否能创建/写入）
- 若在公网映射环境运行，避免把附件下载链路绑定临时外网域名，优先走本机 API 回源。
