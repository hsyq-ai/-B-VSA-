# 红智秘书页面 Bug 排查记录

---

## 一、想要实现的效果

### "首次登录触发，后续静默恢复"的秘书进入规则

#### 场景 1: 首次登录

- 用户刚登录成功后，系统应该**自动跳转到红智秘书页面**
- 这一次进入要**主动唤醒红智秘书**
- 可以显示"正在唤醒中"这类状态
- 然后生成首次欢迎和开场

#### 场景 2: 同一登录态内后续操作

- 用户点去别的导航，再回到红智秘书
- 这时**不能再次唤醒**
- 不能重新欢迎
- 不能重新初始化
- **只能恢复原来的秘书会话内容**

#### 场景 3: 重新登录

- 退出账号后再次登录
- 这又算一次新的"首次登录"
- 应该再次自动跳转到红智秘书，并重新唤醒一次

### 总结

| 场景 | 行为 |
|-----|------|
| 首次登录 | 自动跳转 + 唤醒秘书 + 生成欢迎 |
| 同一登录态内返回 | 静默恢复原会话内容，不做任何初始化 |
| 重新登录 | 视为首次登录，重新执行唤醒流程 |

## 二、目前存在的问题现状

### Bug 1: 返回秘书页面时会话内容丢失

**现象描述**：

- 首次登录：✅ 正常工作（跳转到秘书页面，唤醒秘书）
- 同一登录态内返回：❌ 显示"正在唤醒中..."状态，会话内容丢失

**用户操作路径**：

```
登录 → 自动跳转到红智秘书 → 显示"正在唤醒中" → AI 响应正常
   ↓
点击导航去其他页面（如"工作台"、"文件"等）
   ↓
点击导航返回"红智秘书"
   ↓
❌ 问题出现：显示"正在唤醒中..."，之前的会话内容消失了
```

### Bug 2: 首次登录后"正在唤醒中"状态永远不消失

**现象描述**：

- 首次登录后，秘书唤醒流程启动
- "正在唤醒中..."状态一直显示
- AI 没有任何响应
- 等待再久也不会有结果

**用户操作路径**：

```
登录 → 自动跳转到红智秘书 → 显示"正在唤醒中..." → 永远卡在这里
                                            ↓
                                    ❌ AI 无响应
```

### 跨 iframe 通信机制

父页面通过 `CustomEvent` 与 iframe 内的 Chat 组件通信：

```typescript
// 父页面发送隐藏提示词
const emitHiddenSubmit = (sessionId, prompt, processingText) => {
  const targetWindow = iframe.contentWindow;
  targetWindow.currentSessionId = sessionId;  // 关键：设置 sessionId
  targetWindow.dispatchEvent(new CustomEvent("copaw-hidden-submit", {
    detail: { id: sessionId, prompt, processingText }
  }));
};

// 子页面监听并处理
window.addEventListener("copaw-hidden-submit", (e) => {
  pendingSceneRef.current = { id, prompt, ... };
  schedulePendingSceneSubmit(0);  // 延迟发送，等待 chatInput 就绪
});
```

## 四、Bug 1 深度分析：返回时会话内容丢失

### 已发现的问题

#### 问题 1: React 状态丢失

当用户从秘书页面离开再返回时，组件会卸载再重新挂载：

| 状态 | 初始值 | 挂载后 |
|-----|-------|--------|
| `chatSrc` | `useState("")` | 空字符串 → iframe 不渲染 |
| `secretarySessionIdRef.current` | `useRef(null)` | null |
| `secretaryBootstrappedEpochRef.current` | 从 sessionStorage 恢复 | 正确的 epoch |

#### 问题 2: useEffect 提前返回

```typescript
// Secretary.tsx line 514-530
useEffect(() => {
  const loginEpoch = getStoredLoginEpoch();
  if (secretaryBootstrappedEpochRef.current === loginEpoch) return;  // ← 直接返回！
  
  // 以下代码不会执行
  const existingId = await findExistingSecretarySession(loginEpoch);
  if (existingId) {
    setSecretarySession(existingId);  // ← chatSrc 不会被设置
  }
}, []);
```

因为 `secretaryBootstrappedEpochRef` 正确恢复了 epoch 值，useEffect 直接返回，**没有恢复 `chatSrc`**。

#### 问题 3: chatSrc 状态未恢复

`secretarySessionStorageKey` 中存储了会话 ID，但组件挂载时从未读取这个值来恢复 `chatSrc`。

### 相关代码位置

- `Secretary.tsx` line 100-127: 状态初始化
- `Secretary.tsx` line 514-530: useEffect 检查登录 epoch
- `Secretary.tsx` line 165-184: `setSecretarySession` 函数

## 五、Bug 2 深度分析："正在唤醒中"永远不消失

### 状态流转分析

#### 设置"唤醒中"的路径


#### 清除"唤醒中"的路径

```typescript
// Secretary.tsx line 532-540
useEffect(() => {
  if (!chatFrameReady) return;  // 需要 iframe 加载完成
  const text = pendingProcessingStatusRef.current;
```

**注意**: 这只清除了父页面的状态，**iframe 内部的"正在处理"卡片需要 Chat 组件自己清除**。

#### iframe 内部的处理逻辑

```typescript
  pendingSceneRef.current = { id, prompt, processingText, ... };

const schedulePendingSceneSubmit = (delay) => {
  // 检查 1: currentSessionId 是否匹配
  const currentId = window.currentSessionId || chatIdRef.current;
  if (pending.id !== currentId) {
    pending.attempts++;
    if (pending.attempts <= 30) schedulePendingSceneSubmit(200);  // 重试
    else hideProcessingCard(0);  // 放弃
    return;
  }
  
  // 检查 2: chatInput 是否就绪
  const inputReady = findChatInput();
  if (!inputReady) {
    pending.attempts++;
    if (pending.attempts <= 40) schedulePendingSceneSubmit(200);  // 重试
    else hideProcessingCard(0);  // 放弃
    return;
  }
  
  // 发送隐藏提示词
  document.dispatchEvent(new CustomEvent("handleSubmit", { ... }));
};
```

### 可能的卡住原因

#### 疑点 1: sessionId 不匹配

```typescript
// 父页面设置
targetWindow.currentSessionId = sessionId;  // emitHiddenSubmit 中

// 子页面检查
const currentId = window.currentSessionId || chatIdRef.current;
```

如果 `chatIdRef.current`（来自 URL 参数）与 `pending.id`（来自事件）不一致，会一直重试。

#### 疑点 2: chatInput 未就绪

```typescript
const findChatInput = () =>
  document.querySelector('.copaw-chat-anywhere-input textarea, ...');
```

如果 AgentScopeRuntimeWebUI 的 chatInput 组件使用了不同的 CSS 类名，`findChatInput()` 永远返回 null。

#### 疑点 3: scene_prompt 双重触发

```typescript
// Chat/index.tsx line 1395-1420: 自动执行 scene_prompt
useEffect(() => {
  if (sceneAutoRunRef.current[chatId]) return;
  const prompt = meta?.scene_prompt;
  if (!prompt) return;
  sceneAutoRunRef.current[chatId] = true;
  pendingSceneRef.current = { id: chatId, prompt, ... };
  schedulePendingSceneSubmit(200);
}, [chatId]);
```

同时还有 `copaw-hidden-submit` 事件也会触发 `schedulePendingSceneSubmit`，可能导致冲突。

#### 疑点 4: lastSubmittedSceneKey 去重

```typescript
// Chat/index.tsx line 983-991
const lastSubmitted = lastSubmittedSceneKeyRef.current;
if (lastSubmitted && lastSubmitted.key === pendingKey && Date.now() - lastSubmitted.ts < 8000) {
  pendingSceneRef.current = null;
  return;  // 直接返回，不发送
}
```

如果 8 秒内收到相同 key 的请求，会被忽略。
