## 1. 当前前端设计方式（现状基线）

## 1.1 架构形态
当前为“**单应用壳层 + 双业务域 + 聊天内核复用**”模式：

1. **统一壳层**：`layouts/MainLayout/index.tsx`
   - 统一登录态、路由守卫、全局轮询、布局切换。
2. **双业务域**：
   - 员工端：`/app/*`
   - 管理端：`/admin/*`
3. **聊天内核复用**：`pages/Chat/index.tsx`
   - 普通页面与 embed 页面共用同一套聊天能力。

## 1.2 页面组织特点
1. 员工端工作流入口聚合在 `EmployeeSidebar.tsx`（动态菜单/场景化入口）。
2. `Secretary.tsx` 负责“红智秘书”业务编排，并通过 iframe 嵌入聊天页。
3. `Sessions.tsx` 负责会话中心展示与管理。
4. `sessionApi/index.ts` 实际承担前端“会话与推送中枢”职责。

## 1.3 状态管理方式
当前主要采用：
- React Hooks（`useState/useEffect/useRef/useMemo`）
- `sessionApi` 单例 + `localStorage/sessionStorage`
- `window.CustomEvent` 事件驱动（如推送会话更新、隐藏提交）

结论：当前架构可持续演进，但应减少页面内隐式状态与字符串事件耦合。

## 2. 升级目标（与当前业务融合）

## 2.1 业务目标
1. **秘书代办极简党务**：支持一句话发起“三会一课/通知/备案/纪要”等流程闭环。
2. **组织关怀与思想动态感知**：基于分身与会话行为形成关怀提醒与组织温度看板。
3. **活动分身协同**：活动通知、报名回执、提醒催办、结果回流全链路自动化。
4. **伴随式思政辅导**：在工作与写作场景中提供政策依据、党史知识与个性化学习推荐。
5. **党员测评**：支持 `优秀/合格/一般` 三档测评，输出优秀党员候选。
6. **支部评比**：支持支部横向评比，支部数 `>=3` 时支持优秀支部评选，并提供选人提拔辅助。
7. **指示精神时效响应**：快速形成“政策解读 -> 企业分析 -> 落地行动项 -> 文档产出”。
8. **秘书统一入口不变**：所有能力优先通过红智秘书和工作台承载。

## 2.2 技术目标
1. 形成可扩展的“党建业务前端模块层”。
2. 建立统一任务状态模型（会话、消息、待办、评比流程一致化）。
3. 降低 `Chat` 与 `Secretary` 巨组件复杂度。
4. 为后续实时化（WebSocket/SSE）预留通道。

## 3. 目标技术架构（前端）

## 3.1 分层建议

1. **壳层（Shell）**
   - 文件：`layouts/MainLayout/index.tsx`
   - 职责：登录态、路由、全局消息通道、全局通知。

2. **业务层（Feature Pages）**
   - 员工侧：`pages/Employee/*`
   - 管理侧：`pages/*`
   - 职责：场景页面编排、表单/看板交互。

3. **会话层（Conversation Core）**
   - 文件：`pages/Chat/index.tsx` + `pages/Chat/sessionApi/index.ts`
   - 职责：会话生命周期、推送合并、场景启动、embed 通信。

4. **能力层（Domain Modules，新增）**
   - 新目录建议：`features/party/*`
   - 职责：党务代办、组织关怀、活动协同、思政辅导、党员测评、支部评比、指示响应的页面组件、hooks、类型与API适配。

## 3.2 路由扩展建议
在 `MainLayout` 增加以下员工端路由（建议）：
- `/app/party/party-affairs`
- `/app/party/organization-care`
- `/app/party/activity-collab`
- `/app/party/learning-coach`
- `/app/party/member-evaluation`
- `/app/party/branch-ranking`
- `/app/party/directive-center`

说明：入口仍可由 `EmployeeSidebar` 场景卡跳转，也可通过秘书指令直达。

## 4. 关键模块升级方案

## 4.1 `EmployeeSidebar.tsx`（入口治理）

### 升级动作
1. 新增党建功能组：
   - 党务代办中心
   - 组织关怀中心
   - 活动协同中心
   - 思政辅导中心
   - 党员测评
   - 支部评比
   - 指示精神响应中心
2. 保持动态配置机制（dashboard/expert/prompt templates）。
3. 增加角色控制（书记/支委/党员/观察角色）。

### 交付标准
- 不同角色看到不同菜单与默认动作。
- 菜单点击后可写入场景参数并跳转工作台或对应业务页。

## 4.2 `Secretary.tsx`（秘书编排）

### 升级动作
1. 增加党建指令模板分发：
   - 发起三会一课/组织生活/党务通知与纪要
   - 发起活动报名回执与自动催办
   - 发起组织关怀与思想动态跟进
   - 发起伴随式思政学习任务
   - 发起党员测评
   - 生成支部评比任务
   - 请求政策解读/企业分析文档
2. 保持“单登录周期单秘书会话”策略，避免重复唤醒。
3. 增加党建任务卡区块（流程状态可视化）。

### 交付标准
- 同一秘书会话可连续承载多类党建任务。
- 任务执行结果可追溯到会话与文档。

## 4.3 `Chat/index.tsx`（聊天内核拆分）

### 升级动作
建议按职责拆分子模块（不改变行为）：
1. `sceneBootstrap`：场景启动与隐藏提示处理。
2. `embedBridge`：iframe/CustomEvent 通信。
3. `streamProcessor`：流式响应解析。
4. `attachmentPipeline`：附件上传与上下文注入。

### 交付标准
- Chat 主文件规模显著下降，功能回归不受影响。
- 场景扩展无需再在单文件追加大量 if/else。

## 4.4 `sessionApi/index.ts`（会话与推送中枢）

### 升级动作
1. 引入明确类型：`SessionMeta`, `PushMeta`, `PartyTaskMeta`。
2. 统一 `id/realId/sessionId` 映射规则与校验函数。
3. 对外提供党建流程辅助方法：
   - `upsertPartySessionMeta`
   - `appendPartyTaskEvent`
4. 为实时通道预留接口（替代固定轮询依赖）。

### 交付标准
- 会话恢复稳定，避免临时 ID 误请求后端。
- 推送/消息/评比流程元数据可统一查询。

## 4.5 `Sessions.tsx`（会话中心）

### 升级动作
1. 增加党建筛选维度：党务/关怀/活动/学习/测评/评比/指示。
2. 支持“流程状态列”（待办、执行中、已完成、待复核）。
3. 增加“快速跳转业务页”操作（查看测评、查看支部榜单、查看报告）。

### 交付标准
- 会话中心从“消息列表”升级为“业务流程索引”。

## 5. 新增功能模块设计（建议）

## 5.1 党员测评模块（`member-evaluation`）

### 页面能力
1. 测评发起：选择范围/周期/规则。
2. 测评录入：优秀、合格、一般。
3. 候选生成：优秀党员候选自动排序。
4. 结果公示：可导出、可追溯。

### 关键前端数据结构（建议）
- `MemberEvaluationRecord`
  - `memberId`
  - `grade: '优秀' | '合格' | '一般'`
  - `score`
  - `evidenceRefs[]`
  - `reviewStatus`

## 5.2 支部评比模块（`branch-ranking`）

### 页面能力
1. 支部指标配置（权重、周期、数据源）。
2. 支部排名看板。
3. 当支部数 `>=3` 自动启用“优秀支部评选”。
4. 选人提拔辅助页（候选画像+证据链）。

### 关键前端数据结构（建议）
- `BranchRankingRecord`
  - `branchId`
  - `totalScore`
  - `dimensionScores`
  - `excellentBranchEligible`
  - `talentCandidates[]`

## 5.3 指示精神响应中心（`directive-center`）

### 页面能力
1. 最新指示列表（来源、发布时间、状态）。

4. 文档中心（日报/周报/月报/专题报告）。

### 关键前端数据结构（建议）

  - `directiveId`

## 5.4 党务代办模块（`party-affairs`）

## 5.5 组织关怀模块（`organization-care`）

## 5.6 活动协同模块（`activity-collab`）

2. 阶段2：引入 SSE/WebSocket 通道，仅替换消息推送链路。
3. 阶段3：推送与会话事件统一到 typed event bus。

## 7. 开发规范（本项目适配）

1. **不破坏现有秘书单会话机制**（登录周期内固定会话）。
2. **避免在页面堆叠业务逻辑**：复杂逻辑下沉 `hooks/services/features`。
3. **事件命名与类型化**：禁止散落匿名字符串事件。
4. **统一时间与触发语义**：会话触发时间、测评时间、评比时间统一格式。
5. **先扩展、少重写**：优先增量改造，减少全量重构风险。

## 8. 分阶段实施计划（建议）

## 阶段A（1-2周）：结构搭建
- 建立 `features/party` 目录与基础类型。
- 新增七类 API 模块。
- 完成路由与侧边栏入口接入。

## 阶段B（2-3周）：业务闭环
- 上线党务代办 MVP（会议/通知/纪要流程）。
- 上线活动协同 MVP（通知->回执->催办->汇总）。
- 上线组织关怀 MVP（基础信号 + 跟进闭环）。
- 上线党员测评 MVP（优秀/合格/一般 + 候选输出）。
- 上线支部评比 MVP（>=3 支部触发优秀支部评选）。
- 上线指示中心 MVP（企业画像 + 报告生成入口）。

## 阶段C（2周）：体验与稳定性
- Chat/Secretary/sessionApi 关键逻辑拆分与类型化。
- 会话中心升级为流程索引中心。
- 回归验证秘书会话恢复、推送追加、流程留痕。

## 阶段D（持续）：实时化与治理
- 推送链路从轮询升级到 SSE/WebSocket。
- 补齐审计可回放、评比复议、结果申诉前端流程。
- 完成企业特征化文档全链路自动化（生成、审阅、归档）。
