## 1. 文档目的

这份文档不是产品宣讲材料，而是面向当前代码与系统现状的**升级实施指导文档**。

目标是：

1. 把两份产品方案中的业务方向，转换成当前系统可执行的技术路线。
2. 明确当前系统哪些能力已经具备、哪些能力要保留、哪些结构必须重构。
3. 指导你在**不推倒重来**的前提下，把当前红智系统升级为：
   - 面向政企/央国企/院校的私有化 AI 协作执行平台；
   - 以“红智秘书 + 数字员工 + 书记驾驶舱 + 科研/平台能力复用”为核心形态；
   - 既能承接党建场景，又能保留当前 Agent OS 的科研与平台能力沉淀。

## 2. 先给结论：升级应该怎么做

### 2.1 产品优先级结论

基于两份产品方案，红智系统的产品优先级应收敛为：

1. **AI 提效与突发事件处置**：最高优先级。
2. **极简流程代办**：通过红智秘书把复杂表单流程改造成一句话发起。
3. **绩效考核与组织关怀**：形成对领导侧有价值的数据分析与关怀提醒。
4. **伴随式党建学习与活动协同**：从人工催办升级为分身自动协同。
5. **深度党建扩展模块**：廉政、纪检、工会等按客户需求增购，不应作为第一阶段主线。

### 2.2 技术优先级结论

当前系统升级顺序建议为：

1. **先做架构收敛**，不是先堆新页面。
2. **先做高频场景闭环**，不是先做大而全党建栏目。
3. **先把秘书、会话、任务、审计、推送打通**，再做复杂多智能体编排。
4. **先保留单体私有化部署优势**，不要过早拆成微服务。
5. **先统一 Web 核心**，再扩展 Tauri 桌面端和移动端。

### 2.3 当前系统的关键判断

当前红智系统已经不是空白系统，而是一个已经具备雏形的 Agent OS：

- 已有 `SO + PIA` 双层 Agent 协作模型；
- 已有 `红智秘书`、`会话中心`、`党建场景页`、`科研任务页`；
- 已有 `Agent OS 路由`、`审计链`、`推送队列`、`场景启动`、`本地持久化`；
- 已有 `私有化部署`、`HTTPS 启动`、`桌面容器`、`同源前后端托管` 基础。

因此，升级路线不应是“重写一个新系统”，而应是：

**把当前系统从“场景原型 + 单文件堆叠”升级为“产品化域模型 + 稳定编排内核 + 多端交付壳层”。**

## 3. 两份产品方案转换成技术要求

## 3.1 从 `20260402_产品方向同步.pdf` 得出的技术要求

该文档传达出的核心方向可以转成以下技术要求：

1. **不是 C 端工具，不是传统 OA，而是 B 端私有化部署的智能党建平台**。
   - 技术要求：坚持私有化部署、同源 Web 平台、账号统一、审计与权限隔离。

2. **工作效率类功能优先于党建活动类功能**。
   - 技术要求：第一阶段优先做秘书代办、任务分发、指示响应、突发事件处置，不要先把活动策划做得很重。

3. **AI 必须是核心，不是点缀**。
   - 技术要求：秘书、分身、任务路由、自动回执、指示解读、报告生成必须进入主流程，而不是挂件功能。

4. **APP 是必须做的，而且和 PC 必须是一套系统的两个入口**。
   - 技术要求：前后端 API、登录态、消息、会话、任务、权限模型必须天然支持多端复用。

5. **私有化部署是核心谈判条件**。
   - 技术要求：保持“单客户可独立部署”的结构，避免对公网 SaaS 或第三方强依赖。

6. **外部 C 端 AI 助理不可依赖，能力必须内化到红智系统里**。
   - 技术要求：统一用“红智秘书 / 数字员工 / 数字专家”承接能力，不把关键业务能力建立在外部消费级 AI 工具之上。

## 3.2 从 `项目发展方向总结报告.pdf` 得出的技术要求

该文档传达出的核心方向可以转成以下技术要求：

1. **红智系统不是 OA，而是主动感知、自主协同、深度记忆的数字神经中枢**。
   - 技术要求：系统核心不是表单，而是“会话 + 任务 + 路由 + 审计 + 记忆”。

2. **统一入口必须是红智秘书**。
   - 技术要求：所有高频业务优先走秘书入口，通过技能包和场景模板触发后端服务。

3. **私有化部署路线必须坚持**。
   - 技术要求：保留当前单体部署优势，同时逐步增强 RBAC、敏感动作确认、审计追踪。

4. **架构应朝“Web 内核 + 本地容器 + 移动端”发展**。
   - 技术要求：当前 Web 核心继续作为唯一业务核心，桌面端从现有 `pywebview` 演进到更强容器，移动端采用 API 复用方案。

5. **持续优化 Markdown 事件总线、模型路由、记忆档案系统**。
   - 技术要求：当前事件推送、会话恢复、元数据、记忆与技能学习机制要从“可用”升级到“产品稳定”。

## 5. 当前系统必须保留的能力

升级时，以下能力不应被推翻：

### 5.1 保留统一秘书入口

红智秘书已经是当前系统最自然的统一入口。后续升级不应回到“先点菜单、再填表单”的旧思路。

正确方向是：

- 菜单用于进入场景；
- 真正的高频操作仍通过秘书发起；
- 页面承担“结构化结果展示 + 二次确认 + 审批留痕”的职责。

### 5.2 保留 Agent OS 双层协作模型

当前 `SO + PIA` 结构，是红智系统区别于普通党建平台和 OA 的核心。

后续升级应继续围绕：

- `SO` 负责跨人、跨部门、跨场景编排；
- `PIA` 负责个人上下文、个人回执、个人执行与授权动作；
- 会话与审计贯穿整个流程。

### 5.3 保留党建与科研共用底座

当前代码已经证明：

- 党建场景页；
- 科研实验页；
- 专家中心；
- 技能演化服务；

可以共用同一套：

- 会话内核；
- 任务状态；
- 推送与审计；
- 场景启动；
- 记忆沉淀。

因此升级时不应把系统收缩成“只剩党建”，而应保持：

**党建是商业切入口，科研/数字专家/平台学习是底层能力资产。**

### 5.4 保留单体私有化部署优势

当前单体结构虽然不够优雅，但对私有化交付非常友好：

- 部署简单；
- 便于客户内网落地；
- 便于单客户定制；
- 便于运维与演示；
- 便于快速试点。

在没有真正的性能瓶颈前，不建议先拆微服务。

## 6. 当前系统必须改的地方

## 6.1 前端问题

### 6.1.1 巨型文件过多

当前明显存在以下“巨型文件”问题：

- `Secretary.tsx`
- `Chat/index.tsx`
- `sessionApi/index.ts`
- `EmployeeSidebar.tsx`

问题不是代码多，而是：

- 场景、UI、状态、协议、缓存、轮询、桥接耦合在一起；
- 改一个场景容易影响全局；
- 以后再增加党建/移动端/多角色逻辑会迅速失控。

### 6.1.2 党建域页面已存在，但仍偏“场景页拼接”

当前党建页面可演示，但还没有形成真正的前端域模型：

- API 类型定义分散；
- 各页面对状态、筛选、流程语义的定义不完全统一；
- 仍大量依赖页面内部状态与本地 fallback。

### 6.1.3 推送与实时性仍主要依赖轮询

当前存在多处 `setInterval` 轮询：

- 主布局轮询消息；
- 秘书页轮询消息；
- 科研页轮询任务进度；
- 局部组件轮询状态。

这在单机 MVP 没问题，但在产品化阶段会带来：

- 前端状态抖动；
- 多端同步不一致；
- 服务器无效请求增多；
- 移动端体验不佳。

## 6.2 后端问题

### 6.2.1 `party.py` 承担了过多职责

当前 `party.py` 已经同时承载：

- 党务事务
- 活动协同
- 党员测评
- 支部评比
- 指示直达
- 组织关怀
- 思政辅导

这对 MVP 是高效的，但对后续升级不利：

- 路由层和业务层耦合；
- 一个模块出问题会影响整个党建域；
- 很难做权限、流程、审计规则细化；
- 很难做客户级功能裁剪。

### 6.2.2 党建数据模型仍偏原型期

当前 `party_work_store.py` 的模式是：

- 每个模块一张表；
- 业务载荷主要放 `payload JSON`；
- 公共字段较少。

优点是开发快；缺点是：

- 不利于复杂分析；
- 不利于统计报表；
- 不利于规则版本化；
- 不利于跨模块统一治理。

### 6.2.3 推送链路仍偏单机实现

当前推送链路依赖：

- SQLite 队列；
- 文件事件总线/消息队列；
- 前端轮询拉取；
- 本地审计表。

这非常适合单实例私有化，但如果后续要支持：

- 多端同步；
- 更强实时性；
- 更细粒度流程事件；
- 更完整运维监控；

就需要抽象成更稳定的事件总线接口。

## 6.3 多端与容器问题

### 6.3.1 桌面端当前是 `pywebview`，不是最终形态

当前桌面端已可运行，但更适合演示或轻量桌面封装，不适合作为长期政企桌面方案的最终形态。

### 6.3.2 移动端还没有真正落地

产品方案明确要求：

- 领导和员工需要手机端可用；
- PC 与手机必须账号统一、数据同步、同一系统。

这意味着当前系统必须进入“多端准备态”。

## 6.4 安全与合规问题

当前系统已经有基础权限和审计，但仍缺少更完整的：

- 角色分层授权；
- 跨用户敏感动作二次确认；
- 政治内容安全策略；
- 评优评先规则版本化；
- 指示精神来源与版本追踪；
- 多级组织数据隔离。

## 7. 升级总体架构建议

建议将红智系统升级为“**一个 OS 内核 + 两条业务主线 + 三个交付端**”的架构。

## 7.1 一个 OS 内核

即保留并强化以下核心底座：

1. 红智秘书入口
2. 会话系统
3. 场景启动机制
4. Agent OS（SO + PIA）
5. 路由与审计
6. 推送与任务事件
7. 记忆与技能演化

这些能力已经在当前代码中存在，应继续沉淀，而不是另起炉灶。

## 7.2 两条业务主线

### 主线 A：党建与组织治理

包括：

- 指示直达
- 事务中心
- 活动协同
- 组织关怀
- 思政辅导
- 党员测评
- 支部评比

### 主线 B：科研与平台能力

包括：

- 科研实验任务
- 数字专家中心
- 平台学习与技能演化
- 知识检索、文档创作、任务自动化

这两条主线不应分裂成两个系统，而应共用同一底层能力。

## 7.3 三个交付端

### 交付端 1：Web 主系统

这是当前必须继续强化的主交付面。

### 交付端 2：桌面端容器

当前保留现有桌面端方案，后续评估升级到更强本地容器。

### 交付端 3：移动端

建议以“同后端 API + 同权限模型 + 同消息机制”为原则，按阶段推进：

- 第一阶段：移动 Web / PWA / H5 适配；
- 第二阶段：小程序或专用 APP；
- 第三阶段：按客户交付要求做国企环境适配。

### 8.1.3 统一前端业务类型

建议统一以下类型定义：

- `TaskMeta`
- `SceneMeta`
- `AuditMeta`
- `PartyTaskMeta`
- `ResearchTaskMeta`
- `PushMeta`

并把这些类型抽到：

```text
console/src/features/core/types/
```

这样可以避免每个页面自己解释 `task_id / trace_id / conversation_key / party_module / status`。

### 8.1.4 路由与入口治理建议

保留现有路由，但做一层场景注册表：

```text
console/src/features/core/scene-registry.ts
```

将当前分散在：

- `EmployeeSidebar.tsx`
- `Secretary.tsx`
- 部分页面的 `sessionStorage` 场景写入

统一收敛，避免后面出现越来越多的场景字符串硬编码。

### 8.1.5 多端适配建议

前端要从现在开始为移动端做准备：

1. 所有高频页面增加 `mobile-first` 布局约束。
2. 秘书页、会话页、事务页优先支持窄屏交互。
3. 将重度桌面页面和移动高频页面分开设计，不要求移动端完全复刻桌面复杂看板。
4. 保证登录态、会话、消息、任务卡、回执链路在移动端可复用。

### 这样拆分的好处

3. 存储层负责表读写。
4. 更方便做客户级能力开关与模块裁剪。

## 8.2.2 建立统一流程编排接口

当前很多场景已经具备“创建 -> 推送 -> 回执 -> 完成”的形态，建议抽象出统一流程接口：

- `create_task()`
- `dispatch_task()`
- `complete_task()`
- `record_audit()`
- `append_conversation_meta()`

这样：

- 党务任务卡
- 活动协同提醒
- 组织关怀跟进
- 科研实验跟进

可以共用同一套编排框架。

## 8.2.3 建立“指示直达”专属智能服务

建议新增：

```text
src/copaw/app/services/party/directive_intelligence_service.py
```

职责：

1. 接收中央或上级指示内容；
2. 结合企业画像进行差距分析；
3. 生成行动项；
4. 派发到对应部门/数字员工；
5. 输出日报/周报/月报模板；
6. 写入审计链与会话元数据。

这会是红智系统对外最强卖点之一。

## 8.2.4 强化 `agent_os.py`，把它做成真正的协作总线

建议继续以 `agent_os.py` 为协作总线入口，不要额外再造一套“党建消息总线”。

升级重点：

1. 场景启动参数标准化；
2. 跨用户授权动作标准化；
3. 目标人解析规则标准化；
4. 审计字段统一；
5. 支持更稳定的任务卡回执模型。

## 8.2.5 将 Markdown 事件总线抽象为接口，而不是马上替换

产品方案中提到继续优化 Markdown 事件总线，这是合理的。

建议做法不是立刻废弃，而是：

1. 当前阶段保留现有兼容机制；
2. 新增统一事件总线接口，例如：
   - `publish_event()`
   - `consume_event()`
   - `ack_event()`
3. 先让现有 SQLite / markdown / 本地队列实现该接口；
4. 后续如客户有更高并发要求，再切 Redis / NATS / Kafka 等实现。

## 8.3 数据层升级建议

## 8.3.1 当前数据层结论

当前数据层非常适合 MVP：

- `party_work_store.py`：轻量、快速、易改；
- `message_queue_store.py`：适合单实例内网部署；
- `research_experiment_store.py`：已经能承接科研任务状态；
- 工作目录 + 记忆目录：对私有化落地很友好。

但产品化后，需要做结构升级。

## 8.3.2 数据层升级方向

### 第一阶段：保持 SQLite，不换数据库

先做：

- 统一表字段；
- 增加查询索引；
- 增加审计字段；
- 增加状态字段版本；
- 规范 `department / owner / task_id / trace_id / conversation_key / biz_domain / module`。

### 第二阶段：增加报表/分析友好的结构化字段

当前 `payload JSON` 模式建议保留，但要补充结构化字段，尤其是：

- `status`
- `owner_user_id`
- `department`
- `task_type`
- `priority`
- `due_at`
- `last_push_at`
- `result_status`

### 第三阶段：按客户规模决定是否升级 PostgreSQL

当出现以下情况时再考虑升级：

- 单实例数据量明显增大；
- 报表统计变复杂；
- 并发写入变多；
- 需要更强事务与多端一致性。

否则在政企私有化场景下，SQLite 仍然是高性价比方案。

## 8.3.3 记忆与档案升级方向

建议新增“业务场景标签 + 生命周期”机制：

- 组织记忆
- 党建知识记忆
- 科研知识记忆
- 用户画像记忆
- 指示精神记忆
- 关怀观察记忆

原则：

1. 不是所有会话都应该永久沉淀；
2. 记忆必须可追溯来源；
3. 敏感画像必须受权限控制；
4. 指示类内容必须带版本和时间戳。

---

## 8.4 安全、权限与合规升级建议

这部分是政企客户能否真正采购的关键。

## 8.4.1 权限模型建议

从当前“用户登录 + 基本角色控制”，升级为：

- 超级管理员
- 组织管理员
- 书记/支委
- 普通党员/员工
- 观察员/审计员
- 数字员工代理权限

每类角色要明确：

- 可查看的数据域；
- 可发起的任务类型；
- 可授权的自动动作；
- 是否需要二次确认；
- 是否可导出报告。

## 8.4.2 敏感动作二次确认

以下动作必须增加确认机制：

- 跨用户任务下发；
- 对外通知；
- 自动写入正式报告；
- 自动完成他人回执；
- 涉及本地文件写入或敏感数据汇总。

## 8.4.3 审计要求

建议所有关键动作都形成统一审计记录：

- 谁发起；
- 谁执行；
- 由哪个场景触发；
- 对应哪个会话；
- 对应哪个 `task_id`；
- 对应哪个 `trace_id`；
- 是否成功；
- 是否被人工确认。

## 8.4.4 评优评先公平性要求

党员测评、支部评比属于高敏感业务，必须做到：

1. 评分规则版本化；
2. 每次评分保留依据；
3. 候选推荐可解释；
4. 公示与复核链路可追踪；
5. 禁止仅凭大模型主观印象直接输出结论。

## 9. 多端升级建议

## 9.1 Web 端：继续作为唯一业务核心

当前 Web 架构是最成熟的，应继续作为：

- 所有业务能力的源实现；
- 所有接口契约的定义源；
- 所有工作流的标准入口。

## 9.2 桌面端：当前先保留，后续升级容器

当前桌面端基于 `pywebview`，建议策略：

### 近期
- 保留现有桌面能力用于演示和内部交付；
- 不在第一阶段重写桌面壳。

### 中期
- 评估是否迁移到更强桌面容器；
- 如需更强本地权限、系统托盘、离线包、通知能力，再考虑切换。

结论：

**不要在业务域还没稳定时先做桌面容器重构。**

## 9.3 移动端：必须进入产品路线

建议分两步：

### 第一步：移动 Web / PWA

优先支持：

- 登录
- 红智秘书
- 会话消息
- 指示直达摘要
- 活动确认/回执
- 简单审批与提醒

### 第二步：小程序或专用 APP

当有明确客户要求时，再做：

- 推送通知深度集成；
- 二维码学习入口；
- 活动签到；
- 轻量会议参与；
- 组织关怀提醒。

原则：

- PC 和移动必须共用账号、权限、会话、任务、消息模型；
- 不能做成两个割裂系统。

- 推送元数据

## 13. 验收标准建议

2. 红智秘书承接高频业务不回退成纯聊天展示。

## 14. 最终建议

红智系统当前最正确的升级方向，不是“做一个更大更全的党建 OA”，而是：

**把现有 Agent OS 继续升级成一个以红智秘书为统一入口、以数字员工为协作主体、以书记驾驶舱为业务展示面、以科研与平台能力为底层资产的私有化智能协作执行系统。**

一句话概括这次升级：

**不是重做，而是收敛；不是堆页面，而是做中枢；不是做功能目录，而是做可执行闭环。**
