# CoPaw v1.6 版本说明

## 🚀 新增功能与改进

### 1. 统一推送消息链路，实现会话追加

**问题描述**：
在 v1.5 版本中，用户通过前端网页发送给 AI 系统的通知消息（例如“通知贺柏鑫明天下午开会”），目标用户（贺柏鑫）虽然能收到“新消息”弹窗提示，但消息内容未能自动追加到与发送方（陈文豪）的聊天会话中，导致消息割裂，无法形成连续对话。

**根因分析**：
经过排查，发现主要问题在于后端推送消息的存储机制不一致：
- 通知 Agent 将消息写入了基于 SQLite 的持久化队列 `MessageQueueStore`。
- 但前端轮询 `/api/messages/pull` 接口时，后端读取的是内存中的 `MESSAGE_QUEUE`。
- 这种读写分离导致前端无法获取到实际的通知消息内容，从而无法将其追加到聊天会话。
- 此外，`event_listener` 中事件总线文件的路径硬编码，存在与 `notification_service` 写入路径不一致的风险。

**解决方案**：
1.  **统一后端推送存储**：
    - 在应用启动时，将 `MessageQueueStore` 实例挂载到 `app.state.message_store`。
    - 改造 `/api/messages/pull`、`/api/messages/debug` 和 `/api/messages/test-push` 接口，使其统一从 `app.state.message_store` 中读取和写入消息。
    - 确保从 `message_store` 拉取的消息包含 `source_user_id` 和 `source_user_name` 等关键信息，以便前端进行会话归并。
2.  **统一 Agent 推送写入**：
    - 确保 `notification_agent` 和 `stats_agent` 都将消息写入 `app.state.message_store`。
3.  **统一事件总线路径**：
    - 将 `event_listener.py` 中 `EVENT_BUS_FILE` 的定义修改为基于 `WORKING_DIR` 的相对路径，与 `notification_service` 保持一致，避免路径错位问题。

**影响**：
- 解决了通知消息无法追加到聊天会话的问题，用户现在可以收到完整的通知内容并形成连续对话。
- 提升了消息推送机制的健壮性和一致性。
- 优化了用户体验，使 AI 助理的通知功能更加完善。

## 🛠️ 技术栈更新

- **前端依赖**：更新了 `npm` 依赖，并重新构建了 `console` 前端。
- **后端依赖**：重新安装了 `pip` 依赖（开发模式）。

## ✅ 验证方式

1.  **基本验证**：
    - 确保 `copaw app` 服务正常启动，并通过 `curl http://127.0.0.1:8088/api/version` 确认版本信息。
2.  **功能验证**：
    - 使用两个不同的用户账号登录前端控制台。
    - 用户 A 向用户 B 发送一条通知消息（例如：“通知用户B 明天下午开会”）。
    - 观察用户 B 是否能收到消息，并且该消息是否自动追加到与用户 A 的聊天会话中。
    - 用户 B 在该会话中回复用户 A，观察消息是否能正确转发并追加到用户 A 的会话中。
