### 1. 改造目标与约束

- **目标**
  - 为当前 CoPaw 前端添加：**登录 / 注册** 能力，未登录不可使用业务功能。
  - 建立 **员工账号体系 + 审批机制**，支持姓名/手机号登录，密码简单即可。
  - 实现 **员工私有记忆 + 公司公共记忆**：
    - 员工之间的私有记忆互相隔离；
    - 模型回答时仅使用“当前员工私有记忆 + 公共记忆”。
  - 提供一个简单的 **后台管理页 `/manager`**：
    - 管理员工账号（审批注册、启停、角色分配、重置密码）；
    - 管理公共档案与个人档案（Markdown）。

- **约束与前提**
  - 后端集成到 **现有 CoPaw 后端工程** 中，不单独新服务。
  - 持久化使用 **SQLite**。
  - 登录态采用 **JWT + localStorage**。
  - **注册需要审批**；未审批不能登录。
  - 不做验证码 / 短信，不做自助找回密码，只支持后台重置密码。

### 2. 角色 & 权限模型

- **角色类型**
  - `employee`（普通员工）：
    - 能登录主控制台；
    - 只能访问：自己的私有记忆 + 公共记忆；
    - 只能查看/编辑自己的个人信息（后期可扩展）。
  - `admin`（管理员）：
    - 拥有 `employee` 的全部能力；
    - 额外可访问 `/manager` 后台：
      - 查看/审批/禁用用户；
      - 修改用户角色（employee/admin）；
      - 重置密码；
      - 管理公共档案；
      - 管理任一员工的私有档案。

- **注册审批流程（最小版）**
  1. 用户在登录页点击“注册”，填写姓名 / 手机号 / 密码。
  2. 后端创建账号，状态为 `pending`（待审批），角色默认为 `employee`。
  3. 前端提示“注册成功，已提交审核，请联系管理员审批后再登录”。
  4. 管理员在 `/manager` 的“用户列表”中，对 `pending` 用户执行：
     - “通过”：状态改为 `active`，可登录；
     - “拒绝”：可删除账号或标记为 `rejected`。
  5. 只有 `status = active` 的账号才允许登录。

- **首个管理员产生方式**
  - **首个注册成功的账号自动变为 `admin` 且 `active`**，用于 bootstrap：
    - 检测条件：如果当前数据库中没有任何 `admin` 账号，第一位注册用户在审批通过时自动赋予 `admin` 角色。
    - 之后的注册账号均默认为 `employee`（即便没有审批，也不会再自动升为 admin）。

### 3. 数据模型设计（SQLite）

> 这里只描述核心字段；具体表结构实现时再补充索引等细节。

#### 3.1 用户表 `users`

- `id`（主键，自增）
- `name`（姓名）
- `phone`（手机号，唯一）
- `password_hash`（密码哈希）
- `role`（枚举：`employee` / `admin`）
- `status`（枚举：`pending` / `active` / `disabled` / `rejected`）
- `profile_id`（外键，指向 `employee_profiles.id`）
- `created_at`
- `updated_at`

**约束与逻辑：**

- 手机号 **唯一**。
- 姓名 **允许重名**。
- 登录时：
  - 若输入是手机号格式：按 `phone` 查；
  - 否则按 `name` 查：
    - 若匹配 1 条：正常登录；
    - 若匹配多条：返回错误“存在重名，请使用手机号登录”。

#### 3.2 员工档案表 `employee_profiles`

- `id`（主键，自增）—— 即 `profile_id`
- `name`（姓名）
- `phone`（手机号，可与 `users` 冗余，用于检索/显示）
- `department`（部门，可为空）
- `position`（职位，可为空）
- `extra_meta`（JSON，可扩展：入职日期、工号等）
- `created_at`
- `updated_at`

**关系：**

- `users.profile_id` → `employee_profiles.id`（一对一）。

#### 3.3 档案 / 记忆元信息表（可选）

> 实际长内容在 Markdown 文件中，这里只做索引、类型、路径信息，便于后台管理和后续扩展。

- 表：`memory_docs`（可选，第一版可以不做全文，只做 metadata）
  - `id`（主键）
  - `owner_type`（`employee` / `public`）
  - `owner_profile_id`（若 `employee`，则为该员工 `profile_id`；若 `public`，则为 `NULL` 或 0）
  - `doc_type`（`long_term` / `daily` 等，起码有 `long_term`）
  - `file_path`（相对于工作目录的路径）
  - `title`（从 Markdown 第一行解析，便于列表展示）
  - `created_at`
  - `updated_at`

第一版如不想增加复杂度，可以先跳过 `memory_docs` 表，**通过约定好的路径直接读写 Markdown 文件**；后台管理界面只做最简单的“编辑单个文件”。

### 4. 记忆 / 档案目录结构设计

#### 4.1 CoPaw 工作目录 & 系统记忆

- 工作目录：当前已有的 `copaw_work`（具体路径依据 CoPaw 的 `WORKING_DIR`）。
- 系统级记忆：保持原有用途，不动现有逻辑：
  - `copaw_work/MEMORY.md`：系统长期记忆（工具经验、项目偏好等）。
  - `copaw_work/memory/YYYY-MM-DD.md`：系统日常记忆（如果有）。

> 员工档案 & 公共档案 **不写进这些文件**。

#### 4.2 公司 & 员工记忆结构（推荐方案 B / 选项 1）

- **公共记忆（公司公共档案）**
  - 目录：`copaw_work/memory/public/`
  - 核心文件：
    - `copaw_work/memory/public/MEMORY.md`
      - 存放公司级长期档案，如：公司简介、组织架构、规章制度、产品信息等。

- **员工私有记忆（个人档案）**
  - 目录：`copaw_work/memory/employees/`
  - 每个员工一个子目录，以 `profile_id` 命名：
    - `copaw_work/memory/employees/<profile_id>/MEMORY.md`
      - 长期个人档案（员工画像、岗位信息、项目经历等）
    - 将来可扩展：
      - `copaw_work/memory/employees/<profile_id>/YYYY-MM-DD.md`（按日期的附加记忆）

- **自动生成策略**
  - 注册审批通过时：
    - 创建 `employee_profiles` 记录；
    - 在 `employees/<profile_id>/` 下自动生成 `MEMORY.md`，内容例如：

      ```markdown
      # 张三的档案

      （在这里记录本员工的个人档案、岗位信息、项目经历等）
      ```

  - 公共 `public/MEMORY.md` 由管理员手动维护，初始内容可以为：

      ```markdown
      # 公司公共档案

      在这里记录公司简介、规章制度、产品信息等公共内容。
      ```

### 5. 鉴权 & 会话机制

#### 5.1 登录态

- 登录成功后，后端签发 **JWT Token**，包含：
  - `user_id`
  - `profile_id`
  - `role`（employee/admin）
  - `status`（确保是 active）
  - 有效期（例如 7 天，可配置）

- 前端：
  - 将 JWT 存在 `localStorage`（键名如 `copaw_token`）。
  - 每个 API 请求通过 `Authorization: Bearer <token>` 发送。

- 后端：
  - 所有需要鉴权的接口从 Token 解出 `user_id / profile_id / role / status`：
    - `status != active` 的请求直接返回 401/403；
    - `role != admin` 的访问 `/manager` 相关接口直接 403。

#### 5.2 记忆访问权限

- **普通员工访问记忆 / 对话时：**
  - 后端根据 Token 拿到 `profile_id`。
  - 构建一个“可访问的记忆路径列表”：
    - 公共：`copaw_work/memory/public/MEMORY.md`（及可能的扩展）
    - 私有：`copaw_work/memory/employees/<profile_id>/...`
  - 记忆搜索 / 检索工具（如现有 `memory_search`）在这个**受限路径集合**内工作。

- **管理员访问后台 / 档案编辑时：**
  - 后端允许其读取/写入任意：
    - `public/MEMORY.md`
    - `employees/<profile_id>/MEMORY.md`

### 6. 后端接口设计（概念级）

> 这里只列出主要接口，不写具体字段细节。

#### 6.1 认证相关

- `POST /api/auth/register`
  - 入参：`name`, `phone`, `password`, `confirm_password`
  - 逻辑：
    - 创建 `employee_profiles`（基本信息）；
    - 创建 `users` 记录，`status = pending`, `role = employee`；
    - 如果当前无任何 admin 用户，则此用户审批通过后将成为 admin（逻辑可放审批接口）。
  - 返回：注册成功 + “待审批”提示。

- `POST /api/auth/login`
  - 入参：`identifier`（姓名或手机号）, `password`
  - 逻辑：
    - 识别 identifier 是手机号还是姓名；
    - 查找用户；若 `status != active`，拒绝；
    - 密码校验；
    - 返回 JWT + 基础用户信息（`name`, `profile_id`, `role` 等）。

- `GET /api/auth/me`
  - 使用 Token 解析当前用户信息。

#### 6.2 后台管理相关（需 `role=admin`）

- `GET /api/admin/users`
  - 查询用户列表，支持状态过滤（pending/active/disabled）。

- `POST /api/admin/users/{user_id}/approve`
  - 将 `status` 从 `pending` 改为 `active`；
  - 若当前系统无 admin，则将该用户角色设为 `admin`。

- `POST /api/admin/users/{user_id}/reject`
  - 标记为 `rejected` 或直接删除（设计时可选其一）。

- `POST /api/admin/users/{user_id}/reset-password`
  - 管理员为该用户设置新密码。

- `POST /api/admin/users/{user_id}/role`
  - 修改角色为 `employee` 或 `admin`（防止误锁死，可以保留至少一个 admin）。

- `GET /api/admin/memory/public`
- `PUT /api/admin/memory/public`
  - 读取 / 保存 `public/MEMORY.md` 内容。

- `GET /api/admin/memory/employee/{profile_id}`
- `PUT /api/admin/memory/employee/{profile_id}`
  - 读取 / 保存对应员工的 `employees/<profile_id>/MEMORY.md`。

#### 6.3 员工自己的档案接口（可选）

- `GET /api/me/profile`
  - 返回员工档案摘要（用于问候语等）。

- `GET /api/me/memory`
- `PUT /api/me/memory`
  - 可允许员工自己维护一部分档案（如果你希望开放），否则只读。

### 7. 前端改造设计（概念级）

#### 7.1 登录 / 注册弹窗

- 入口：
  - App 初始化时，如果没有有效 Token：
    - 不渲染主内容，弹出一个全局 Modal：`登录 / 注册` Tab。
  - 任何未登录用户尝试访问任意业务路由，都会看到这个弹窗。

- 登录 Tab：
  - 字段：账号（姓名或手机号）、密码。
  - 失败提示不泄露过多信息（统一提示“账号或密码错误”）。

- 注册 Tab：
  - 字段：姓名、手机号、密码、确认密码。
  - 成功后提示“已提交审核，请联系管理员审批后再登录”。

#### 7.2 路由守卫

- 包装主布局组件（含 Sidebar / 内容区）为 `AuthLayout`：
  - 检查 Token；
  - 如果没有或无效 → 强制显示登录弹窗。

- `/manager` 路由：
  - 前端检查 `role === 'admin'`，否则重定向到普通界面或显示 403。

#### 7.3 后台管理页 `/manager`

- 模块：
  - **用户管理**：
    - 列表：姓名、手机号、角色、状态、创建时间。
    - 操作：审批（通过/拒绝）、禁用/启用、修改角色、重置密码。
  - **公共档案管理**：
    - 一个 Markdown 编辑器区域，直接编辑 `public/MEMORY.md`。
  - **员工档案管理**：
    - 按用户 / profile 搜索，打开后可编辑该员工 `MEMORY.md`。

### 8. 员工对话时的记忆使用流程（模型侧）

1. 员工 X 登录，获取 Token（含 `profile_id = P`）。
2. 发起对话请求到后端（例如 `/api/chat`）。
3. 后端在构造模型输入前：
   - 根据 `P` 计算可用记忆源：
     - 公共档案：`public/MEMORY.md`
     - 私有档案：`employees/P/MEMORY.md`（及后续扩展的 daily 文件）
   - 使用现有 `memory_search` 工具在这些文件中检索与当前问题相关的片段。
4. 将检索结果拼入 prompt，严格不读取其他员工的任何文件。
5. 模型根据“公共 + 私有”信息回答员工 X。
