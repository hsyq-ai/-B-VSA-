# 推送链路说明

## 一、完整流程

```
event_bus.md 新增任务
    ↓
event_listener (watchdog 监听文件变更)
    ↓
dispatch_event → handle_notification_task / handle_stats_task
    ↓
notification_agent: 解析 "请 @用户名 消息内容" → get_users_by_name → app.state.message_queue[user_id].append(msg)
    ↓
前端 Chat 页面轮询 GET /api/messages/pull (每 3 秒)
    ↓
pull_messages: JWT user_id → 返回 message_queue[user_id] 并清空
    ↓
前端: createSession + navigate → 新聊天框
```

## 二、关键路径

| 环节 | 文件 | 说明 |
|-----|------|------|
| 任务文件 | `copaw_work/event_bus.md` | 格式: `- [ ] @通知Agent: 用户 @来源 请 @目标 消息` |
| 事件监听 | `event_listener.py` | 监听 event_bus.md，分发到 Agent |
| 通知 Agent | `notification_agent.py` | 解析目标用户，推入 message_queue |
| 消息队列 | `_app.py` MESSAGE_QUEUE | 按 user_id 存储，pull 时清空 |
| 前端轮询 | `Chat/index.tsx` | 每 3 秒 fetch /api/messages/pull |
| 静态资源 | `vite build` → `src/copaw/console/` | 后端直接 serve，无需手动复制 |

## 三、user_id 一致性

- **JWT**: `auth.py` 登录时写入 `user_id: user_row["id"]`（users 表 id）
- **notification_agent**: `target_user_id = str(users[0]["id"])`（get_users_by_name 返回的 id）
- **pull_messages**: `user_id = str(current_user.get("user_id"))`
- 三者必须一致，否则收不到推送

## 四、用户下发指令（聊天触发）

用户在聊天框输入以下格式时，会自动写入 event_bus 并触发推送：

- `通知贺柏鑫 让他把今天的工作内容总结下`
- `请@贺柏鑫 整理文档`
- `让@陈文豪 开会`

格式：`(通知|请|让) [@]目标用户名 消息内容`。系统会解析目标用户，写入 `@通知Agent` 任务，event_listener 处理后推送给目标用户。

## 五、常见问题排查

### 1. 没有新聊天框出现

- **前端未轮询**: 必须停留在 `/chat` 或 `/chat/:id` 页面；若档案不完整会跳转到 `/profile`，此时不轮询
- **API 跨域/BASE_URL**: 若前后端不同源（如前端 55607、后端 8088），需在构建时设置 `BASE_URL=http://后端地址`
- **消息已被拉走**: 队列按 user_id 存储，pull 后立即清空；若轮询间隔内被其他请求先拉走，则本次为空
- **任务格式错误**: 必须是 `请 @用户名`（中间可有空格），且用户名需在 auth.db 的 users 表中存在

### 2. 构建产物位置

- Vite 输出: `CoPaw/src/copaw/console/`（index.html、assets/）
- 后端 `_resolve_console_static_dir()` 优先使用 `pkg_dir/console`，即该目录
- **无需手动复制**，`npm run build` 后重启 CoPaw 即可

### 3. 调试接口

- `GET /api/messages/debug`（需登录）: 返回当前用户的 user_id、待拉取消息数，用于验证链路
